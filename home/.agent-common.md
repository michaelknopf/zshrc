# Local Repo Layout

All local clones live under `~/code/github/<org>/<folder>`. When I refer to a repo by a bare name (e.g. "phony", "pybase"), resolve it by checking `~/code/github/savi/<name>` first (that's where most work happens), then the other orgs: `michaelknopf`, `whisprs`, `SigNoz`, `open-telemetry`.

The local folder name often differs from the GitHub repo name — typically the folder drops the `savi-` prefix or shortens further. Examples: `~/code/github/savi/pybase` is `savisec/savi-python-base`, `~/code/github/savi/pypack` is `savisec/savi-pypack`, `~/code/github/savi/phony` is `savisec/savi-phony-svc`, `~/code/github/savi/comp` is `savisec/savi-compose`, `~/code/github/savi/pytools` is `savisec/savi-pytools`. Use these as examples of the pattern; infer other aliases by stripping the `savi-`/`savi-*-svc` affixes, and confirm by reading `.git/config` when uncertain.

**Do not rely on `../<name>` to find sibling repos.** Always resolve repo references via their absolute `~/code/github/<org>/<name>` path. Worktrees may live anywhere, so never assume a worktree's parent directory contains other repos.

# Always Work in a Worktree

**Never make code changes in the primary clone of a repo.** Any change that modifies files — editing source, writing new files, running formatters/codegen, committing — MUST happen inside a worktree, on a branch, never in the root clone and never on `main`. This applies to every change, however small: a one-line fix in a single repo still goes through a worktree. Create the worktree with `savi-git worktree create <branch>` (see `general-principles.md` for the mechanics), make all edits there, and open the PR from that branch.

Read-only work is exempt: investigating, searching, reading code, running tests/linters without modifying files, and answering questions can all happen directly in the primary clone. The moment the task requires modifying a file, switch to a worktree first.

# Notion is our engineering-doc home

Our canonical engineering docs, guides, and standards live in **Notion** (the "Engineering Docs" database), not in any repo. Many are the source of truth for principles that also back Claude Code skills. Two things follow:

- **Reading or fetching a doc:** use the **Notion MCP** (`mcp__notion__*`). Do **not** run the `savi-notion-fetch` CLI yourself — it is a read-only, aggressively-cached build tool that only exists to inline rarely-changing guides into skill bodies at load time (via `!` preprocessing), and running it to read a doc can hand you a stale copy of one you just edited. Our skills invoke it automatically where appropriate; you never need to.
- **Writing or editing a Notion doc:** use the **`savi:notion`** skill. It carries the create/edit mechanics and points at the canonical **AI Doc Writing Guidelines** (page `38388b3416f780e4bf14c31eb5ecf8b9`) and **Engineering Doc Template** (page `38c88b3416f780728a6be3f2b817e16c`). Don't hand-roll Notion-flavored markdown or the create-from-template flow from memory — invoke the skill.

# Markdown Output Formatting

When writing markdown content — including plan files, documentation, PR descriptions, comments, and any other prose output — do NOT insert hard line breaks inside paragraphs. Let paragraphs be single long lines and rely on the renderer (Claude Code's plan preview, GitHub, terminals) to soft-wrap based on actual width.
